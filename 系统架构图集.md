## 2. 数据处理全链路

### 2.1 RSS 文章从抓取到展示

```mermaid
sequenceDiagram
    participant C as Cron (每天 2:00)
    participant S as RSS Scheduler
    participant P as rss-parser
    participant D as SQLite
    participant F as Filter (LLM)
    participant L as LLM API
    participant PL as Pipeline
    participant V as ChromaDB
    participant TN as TelegramNotifier
    participant UI as Web UI

    C->>S: 触发 fetchAllSources()
    S->>S: 查询活跃 RSS 源 (status='active')
    S->>P: 并发解析 RSS URL (max=5)
    P-->>S: 返回 items[]
    S->>S: 标题去重 (title_normalized)
    S->>D: 批量插入 articles
    
    Note over S,D: 异步触发 (随机延迟 0-30min)
    S->>F: filterArticle(articleId)
    F->>L: LLM 调用 (JSON mode)
    L-->>F: 返回评估结果
    F->>D: 更新 filter_status='passed'|'rejected'
    F->>D: 记录 article_filter_logs
    
    alt filter_status = 'passed'
        F->>PL: processArticle(articleId)
        PL->>PL: Stage1: Markdown 生成
        PL->>PL: Stage2: 翻译 (英文→中文)
        PL->>V: Stage3: Embedding + Upsert
        PL->>D: Stage4: 计算相关文章
        PL->>D: process_status='completed'
        
        Note over PL: Fire-and-Forget
        PL->>TN: sendNewArticle()
        TN->>UI: Telegram 推送
        
        PL->>D: 增量刷新相关文章
    end

    UI->>D: 查询 passed 文章列表
    UI->>V: 语义搜索查询
    D-->>UI: 返回文章元数据
    UI->>UI: 渲染页面
```

## 3. 文章处理流水线

### 3.1 四阶段渐进式处理

```mermaid
stateDiagram-v2
    [*] --> Pending: 文章已保存
    
    Pending --> Processing: 调度器触发 / 手动处理
    Processing --> Markdown: Stage 1 开始
    Markdown --> Translate: Markdown 完成
    Translate --> Vector: 翻译完成
    Vector --> Related: 向量化完成
    Related --> Completed: 全部完成
    
    Processing --> Failed: Stage 2 翻译失败<br/>(重试 3 次后)
    Processing --> Failed: 其他阶段失败<br/>(非致命但标记失败)
    Failed --> Pending: 手动重试
    
    state "Stage 1: Markdown" as Markdown {
        [*] --> Generating
        Generating --> [*]: content → markdown_content
    }
    
    state "Stage 2: Translate" as Translate {
        [*] --> Translating
        Translating --> [*]: LLM 翻译 (英文→中文)
        state "语言检测" as Detect {
            [*] --> CheckZh
            CheckZh --> IsEn: 检测到英文
            CheckZh --> IsZh: 检测到中文
            IsEn --> [*]: 触发翻译
            IsZh --> [*]: 跳过翻译
        }
    }
    
    state "Stage 3: Vector" as Vector {
        [*] --> BuildingText
        BuildingText --> Embedding: 构建向量文本
        Embedding --> Upserting: getEmbedding()
        Upserting --> [*]: ChromaDB upsert
    }
    
    state "Stage 4: Related" as Related {
        [*] --> Querying
        Querying --> Filtering: queryVector(topK=50)
        Filtering --> Saving: score > 0.6
        Saving --> [*]: INSERT article_related
    }
```

## 4. 搜索系统架构

### 4.1 四种搜索模式

```mermaid
graph TD
    Q[用户查询] --> SR{SearchService<br/>search()}
    
    SR -->|mode=semantic| SS[Semantic Search<br/>向量相似度]
    SR -->|mode=keyword| KS[Keyword Search<br/>SQL LIKE]
    SR -->|mode=hybrid| HS[Hybrid Search<br/>加权融合]
    SR -->|mode=related| RS[Related Articles<br/>基于文章相似]
    
    SS --> E1[getEmbedding<br/>query → vector]
    E1 --> C1[ChromaDB query<br/>topK=100]
    C1 --> R1[Rerank (可选)]
    R1 --> E1M[Enrich Metadata<br/>JOIN articles]
    
    KS --> KQ[SQL 查询<br/>title/content LIKE]
    KQ --> KE[Enrich Metadata]
    
    HS --> SM[Semantic 分支]
    HS --> KM[Keyword 分支]
    SM --> C2[ChromaDB]
    KM --> KQ2[SQL 查询]
    C2 --> MF[Score Fusion<br/>Weighted Sum<br/>0.7 × sem + 0.3 × kw]
    KQ2 --> MF
    MF --> HEM[Enrich Metadata]
    
    RS --> RA[获取源文章向量]
    RA --> RV[查询相似文章<br/>topK=50]
    RV --> RScore[Score 阈值<br/>>0.6 优先 5 篇<br/>否则最多 3 篇]
    RScore --> RC[读取 Cache (可选)]
    RC --> REnrich[Enrich Metadata]
    
    E1M --> OUT[SearchResults<br/>按 score 排序]
    KE --> OUT
    HEM --> OUT
    REnrich --> OUT
    
    style SS fill:#e3f2fd
    style KS fill:#e8f5e9
    style HS fill:#fff3e0
    style RS fill:#f3e5f5
```

## 9. 故障转移机制

### 9.1 LLM 故障转移流程图

```mermaid
flowchart TD
    A[调用 filterArticle<br/>需要 LLM] --> B[getUserLLMProvider<br/>按优先级查询]
    B --> C{找到配置?}
    
    C -->|是| D[创建 Provider<br/>OpenAI/Gemini]
    D --> E{测试连接}
    
    E -->|成功| F[使用此 Provider<br/>调用 LLM API]
    E -->|失败| G[标记为不可用<br/>继续下一个]
    
    G --> B
    
    C -->|否| H[读取环境变量<br/>OPENAI_API_KEY]
    H --> I{存在?}
    
    I -->|是| J[创建 Default Provider]
    J --> F
    
    I -->|否| K[抛出异常<br/>"No LLM available"]
    
    F --> L{调用成功?}
    L -->|是| M[返回结果]
    L -->|否| N{重试次数<br/>&lt; 3?}
    
    N -->|是| O[等待 exponential<br/>backoff]
    O --> F
    N -->|否| P[返回错误<br/>文章 rejected]
    
    style P fill:#ffcdd2
    style M fill:#c8e6c9
```

## 总结

本图集涵盖了 LIS-RSS 系统的核心架构和数据流程，主要特点：

1. **清晰的分层设计**：前端 → API → 业务逻辑 → 数据访问 → 存储
2. **完整的数据流水线**：采集 → 过滤 → 处理 → 索引 → 搜索 → 推送
3. **健壮的调度系统**：6 个调度器错开执行时间，避免资源竞争
4. **智能的故障转移**：LLM 多配置链式 fallback，搜索自动降级
5. **多租户隔离**：所有数据按 user_id 隔离，向量 Collection 分片
