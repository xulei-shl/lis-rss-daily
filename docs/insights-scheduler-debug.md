# 洞察报告定时任务分析

## 问题描述

用户反馈：定时洞察报告在 4月27日没有执行，但按配置（间隔10天）看起来应该执行。

> 上次运行：`2026-04-16T23:16:30.760Z`
> 当前日期：`2026-04-27`
> 初始预期执行：4月27日 7:15（北京时间）

## 代码逻辑分析

### 定时触发机制

根据提交 `119400689b6e544c2a02c286e41d5feb2a5f64e5`：

1. **Cron 配置**：`15 7 * * *`（每天 7:15 检查）
2. **间隔配置**：`INSIGHTS_INTERVAL_DAYS = 10`
3. **触发条件**：距离上次成功执行已满 10 天

### 日期记录位置

- **表名**：`settings`

## 问题根因

**根因不是“间隔 10 天的逻辑错误”，而是“精确时间戳 + 固定 Cron 检查点”带来的边界效应。**

当前逻辑记录的是“上次成功执行的精确完成时间”，不是“成功执行的日期”：

- 上次成功时间：`2026-04-16T23:16:30.760Z`
- 换算北京时间：`2026-04-17 07:16:30.760 +08:00`
- 满 10 天的最早时刻：`2026-04-27 07:16:30.760 +08:00`

而 Cron 固定在每天 `07:15:00` 检查一次：

- 4月27日检查时刻：`2026-04-27 07:15:00 +08:00`
- 距离真正满 10 天还差：`90.76` 秒

因此，4月27日这次检查会被判定为“还没到 10 天”，不会触发执行；要等到下一次 Cron，也就是 **4月28日 07:15** 才会执行。

## 服务状态补充说明

“后端未运行”这个结论不能直接成立，至少从当前连通性看并不准确：

- `http://10.40.92.18:8007` 可返回 LIS-RSS 登录页
- `http://10.40.92.18:8007/api/daily-summary/insights/latest` 返回 `401`，说明后端 API 路由存在
- `http://10.40.92.18:3000/api/daily-summary/insights/latest` 返回 `404`，说明 `3000` 更像是另一个前端服务，不是该后端 API

如果要确认 **2026-04-27 当时** 是否存在后端短暂未运行，需要继续查看服务器上的应用日志或 systemd/PM2 进程记录，不能只凭端口 `3000` 推断。

## 解决方案

如果业务期望是“第 10 天早上 7:15 必须执行”，当前实现需要调整判断方式，而不是简单重启服务。

可选修正方向：

1. 改成按“自然日”判断，例如比较上海时区下是否已经跨过第 10 天。
2. 保持当前精确时间差逻辑，但把 Cron 时间延后到 `07:17` 或更晚，避开边界时间。
3. 保持代码不变，但接受实际效果是“满足 10 天后的下一次 Cron 才执行”。

如果只是临时补发，可手动触发：

```bash
cd /opt/lis-rss-daily
pnpm run trigger-insights
```

如果需要排查当日服务状态，再检查部署进程：

```bash
systemctl status lis-rss
journalctl -u lis-rss --since "2026-04-27 07:00:00" --until "2026-04-27 07:30:00"
```

## 时间计算

| 项目 | 时间 |
|---------|------|---------|
| 上次成功执行（UTC） | 2026-04-16 23:16:30.760 |
| 上次成功执行（北京时间） | 2026-04-17 07:16:30.760 +08:00 |
| 满 10 天时刻（北京时间） | 2026-04-27 07:16:30.760 +08:00 |
| Cron 检查时刻（北京时间） | 2026-04-27 07:15:00.000 +08:00 |
| 检查结果 | 提前 90.76 秒，未达到触发条件 |
| 下一次可触发 Cron | 2026-04-28 07:15:00.000 +08:00 |

所以，**4月27日 7:15 不执行符合当前代码逻辑**；如果业务希望当天执行，需要修改调度策略或判断规则。
