## 一、项目概述

### 1.1 项目定位

**LIS-RSS Literature Tracker** 是一个面向学术文献追踪的 **全栈智能信息系统**，核心目标是：

> 自动从学术 RSS 订阅源、期刊官网、Google Scholar 等多源抓取文献 → 使用 LLM 进行领域相关性的智能过滤 → 自动翻译 → 向量化存储 → 提供语义检索和相关推荐

这是一个典型的 **AI-Native 应用**，将大语言模型深度集成到数据处理流水线中，而非简单的 API 调用。

### 1.2 目标用户

- **高校图书馆员/情报分析师**：追踪特定学科领域的最新研究动态
- **科研人员/研究生**：自动化收集特定主题文献，节省手动筛选时间
- **学术机构**：建立部门/课题组的文献追踪系统

### 1.3 核心价值主张

| 痛点 | 本系统解决方案 |
|------|----------------|
| RSS 订阅源过多，无法手动筛选 | 自动抓取 + LLM 过滤，通过率通常 10-30% |
| 英文文献阅读成本高 | 自动翻译摘要和标题（LLM 翻译） |
| 传统关键词搜索匹配度低 | 向量语义搜索 + Rerank 重排序 |
| 发现相关文章困难 | 自动计算语义相似度，推荐相关文献 |
| 多源数据格式不一致 | 统一抽象为 Article 模型（RSS/期刊/关键词） |

### 2.1 多源数据采集系统

支持三种独立的数据源，每种来源有不同的爬取策略和调度周期：

#### 2.1.1 RSS 订阅源 (`rss-scheduler.ts`)

**来源类型**：arXiv、Nature、ScienceDirect 等学术 RSS

**抓取流程**：
```typescript
// 每 source 独立 fetch_interval (默认 3600 秒)
// 并发控制: maxConcurrent = 5
// 去重策略: title_normalized 唯一索引

1. 查询活跃源: SELECT * FROM rss_sources WHERE status='active' AND last_fetched_at < now() - fetch_interval
2. 并发抓取: Promise.race + 信号量控制
3. RSS 解析: rss-parser.parseURL() → items[]
4. 标题去重: generateNormalizedTitle(title) → 查重
5. 批量插入: INSERT INTO articles (rss_source_id, title, url, ...)
6. 触发 auto-filter: 随机延迟 0-30 分钟 (staggerDelayMaxMinutes)
```

**数据库表**：
- `rss_sources`：订阅源配置（user_id, url, fetch_interval, status）
- `rss_fetch_logs`：每次抓取的详细日志（成功/失败/文章数）

**关键特性**：
- 首次抓取限制：最多 50 篇（防止首次导入过多）
- 去重：`title_normalized` 小写化 + 去标点 + 去空格，创建唯一索引
- 失败记录：记录 error_message 和 duration_ms

### 2.3 文章处理流水线 (`pipeline.ts`)

这是一个 **4 阶段渐进式处理系统**，具有幂等性和失败重试能力。

#### 2.3.1 完整流程

```
┌──────────────────────────────────────────────────────────┐
│ Pipeline: processArticle(articleId, userId)              │
├──────────────────────────────────────────────────────────┤
│                                                          │
│ 前置检查: filter_status === 'passed' ?                  │
│   是 → 继续                                              │
│   否 → 跳过 (status = 'skipped')                         │
│                                                          │
│  ── Stage 1: Markdown 生成 ──────────────────────────── │
│  条件: stages.markdown !== 'completed'                   │
│  动作: content → toSimpleMarkdown() → markdown_content   │
│  失败: skip, pipeline 继续                              │
│                                                          │
│  ── Stage 2: LLM 翻译 ──────────────────────────────── │
│  条件: stages.translate !== 'completed'                  │
│  语言检测: detectLanguage(title/content)                 │
│    规则: 中文字符 Unicode 范围 [\u4e00-\u9fff]            │
│    或: 英文字母占比 > 60% 且长度 ≥ 10                    │
│  翻译: translateArticleIfNeeded() (LLM 调用)            │
│  重试: exponential backoff (5s → 10s → 20s, max 3 次)  │
│  保存: article_translations (title_zh, summary_zh)      │
│  标记: translationChanged = true → 需要重新向量化       │
│  失败: 继续（原文向量化），不阻塞流程                    │
│                                                          │
│  ── Stage 3: 向量索引 ─────────────────────────────── │
│  条件: stages.vector !== 'completed' OR translationChanged │
│  向量文本: buildVectorText()                             │
│    TITLE: {title_zh || title}                            │
│    SUMMARY: {summary_zh || summary}                      │
│    CONTENT: {markdown_content || content}                 │
│  批处理: batchSize = 32, 同 user 合并                    │
│  Embedding: getEmbeddingsBatch() (OpenAI/Gemini API)    │
│  存储: chroma.upsert(collection=articles_{userId})      │
│  失败: warn（非致命，继续）                               │
│                                                          │
│  ── Stage 4: 相关文章计算 ─────────────────────────── │
│  条件: stages.related !== 'completed'                    │
│  向量搜索: queryVector(embedding, topK=50)              │
│  筛选: score > 0.6 的优先，否则取最多 3 篇               │
│  保存: article_related (article_id, related_article_id) │
│  失败: warn（非致命）                                     │
│                                                          │
│  完成: process_status = 'completed'                      │
│                                                          │
│  ── Fire-and-Forget 异步操作 ───────────────────────── │
│  1. Telegram 新文章推送 (sendNewArticle)                │
│  2. 增量刷新相关文章 (incrementalRefreshRelated)        │
│     → 让新文章出现在老文章的 related 列表中             │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### 2.3.2 状态管理

**`process_stages` JSON 字段**：
```json
{
  "markdown": "completed",
  "translate": "completed",
  "vector": "completed",
  "related": "pending"
}
```

**`process_status` 枚举**：
- `pending`：等待处理
- `processing`：正在处理
- `completed`：全部完成
- `failed`：处理失败（可重试）

**`filter_status` 枚举**：
- `pending`：待过滤
- `passed`：通过过滤
- `rejected`：未通过

#### 2.3.3 重试机制

```typescript
const RetryConfig = {
  maxRetries: 3,
  baseDelay: 5000,           // 5 秒
  backoffMultiplier: 2.0,    // 指数退避
  maxDelay: 60000            // 最大 60 秒
};

// 执行:
attempt 0 → 失败 → sleep(5000)
attempt 1 → 失败 → sleep(10000)
attempt 2 → 失败 → sleep(20000)
attempt 3 → 失败 → throw (pipeline 整体失败)
```

**仅 Stage 2 (翻译) 启用重试**，其他阶段失败不重试（非致命）。

#### 2.4.3 统一搜索服务 (`search-service.ts`)

**四种搜索模式详解**：

| 模式 | 入口方法 | 流程 | 结果数 | Fallback |
|------|----------|------|--------|----------|
| **SEMANTIC** | `semanticSearchOnly()` | Embedding → Chroma → Rerank → enrich | 50 | ❌ |
| **KEYWORD** | `keywordSearchOnly()` | SQL LIKE (title/content 多字段 OR) | 100 | - |
| **HYBRID** | `hybridSearch()` | Semantic + Keyword 加权融合 | 50 | ✅ (Semantic 失败则纯 Keyword) |
| **RELATED** | `computeRelated()` | 基于向量相似度计算相关文章 | 5-10 | ❌ |

**语义搜索流程**：
```
1. getEmbedding(query) → [0.12, -0.04, ...] (1536 维)
2. queryVector(collection, embedding, topK=100)
   → ChromaDB 返回 [{id, distance, metadata}, ...]
3. 转换分数: score = 1 - distance (cosine metric)
4. Rerank (可选):
   - 查询 llm_configs (config_type='rerank')
   - 调用 Cohere / BGE-Reranker API
   - 按 rerank_score 重新排序
5. enrichWithMetadata():
   - JOIN articles + rss_sources/journals/keyword_subscriptions
   - 补充 title, url, published_at, source_origin
6. 返回 SearchResult[] (按 score 降序)
```

**关键词搜索实现**：
```typescript
// 多字段 OR 匹配
SELECT a.*, rss.name as rss_source_name, ...
FROM articles a
LEFT JOIN rss_sources rss ON a.rss_source_id = rss.id
WHERE (rss.user_id = ? OR journal.user_id = ? OR keyword.user_id = ?)
  AND (
    a.title LIKE '%term1%' OR a.markdown_content LIKE '%term1%'
  ) AND (
    a.title LIKE '%term2%' OR a.markdown_content LIKE '%term2%'
  )
ORDER BY a.published_at DESC
LIMIT 100
```

**相关性计算**：
```typescript
function calcRelevance(title: string): number {
  let score = 0;
  if (title.toLowerCase().includes(query)) score += 0.7;  // 完整匹配
  if (title.toLowerCase().startsWith(query)) score += 0.3; // 前缀匹配
  return Math.min(score, 1);
}
```

**混合搜索分数融合**：


#### 2.5.1 LLM 配置优先级链

```typescript
async function getUserLLMProvider(userId: number, taskType: string): Promise<LLMProvider> {
  // 优先级 1: 任务特定配置 (task_type='filter', is_default=0)

  // 优先级 2: 默认配置 (task_type IS NULL, is_default=1)
  const defaultConfig = await db.llmConfigs.find({
    where: { user_id: userId, config_type: 'llm', is_default: 1, enabled: 1 }
  });
  if (defaultConfig) {
    return createProviderFromConfig(defaultConfig);
  }

  // 优先级 3: 环境变量回退
  if (process.env.OPENAI_API_KEY) {
    return createOpenAIProvider(process.env.OPENAI_API_KEY, process.env.OPENAI_BASE_URL);

}
```

```typescript
async function getLLMWithFailover(userId: number, taskType: string): Promise<LLMProvider> {
  const configs = await getOrderedLLMConfigs(userId, taskType);  // 按 priority 排序

  for (const config of configs) {
    try {
      const provider = createProviderFromConfig(config);
      await provider.testConnection();  // 可选：预检

      log.warn({ configId: config.id, error: error.message }, 'LLM 配置不可用，尝试下一个');

}
```

|--------|-----------------|----------|----------|
| **RSSScheduler** | `0 2 * * *` (每天 2:00) | 所有活跃 RSS 源 | maxConcurrent=5 |
| **RelatedScheduler** | `0 3 * * *` (每天 3:00) | 所有 passed/completed 文章 | batch=100 |
| **JournalScheduler** | `15 2 * * 6` (每周六 2:15) | 所有活跃期刊 | 逐期刊串行 |
| **KeywordScheduler** | `15 3 * * 6` (每周六 3:15) | 所有活跃关键词 | 逐关键词串行 |
| **DailySummaryScheduler** | `0 7 * * *` (每天 7:00) | 昨日通过的文章 | 单线程 |

#### 2.6.1 RSS 调度器实现细节

```typescript
class RSSScheduler {
  private isRunning = false;
  private activeTasks: Map<number, Promise<void>> = new Map();

  async start() {
    this.cron = cron.schedule(config.rssFetchSchedule, async () => {
      await this.fetchAllSources();
    });
  }

  private async fetchAllSources() {
    // 1. 查询需要抓取的源
    const sources = await db.rssSources.findActiveDue();
    // WHERE status='active' AND (last_fetched_at IS NULL OR last_fetched_at < now() - fetch_interval)

    // 2. 并发控制 (Promise.race + 信号量)
    const semaphore = new Semaphore(config.rssMaxConcurrent);  // 5
    const tasks = sources.map(source => semaphore.acquire().then(() =>
      this.fetchSource(source).catch(err => this.handleFetchError(source, err))
    ));

    // 3. 等待全部完成
    await Promise.allSettled(tasks);
  }

  private async fetchSource(source: RSSSource) {
    const startTime = Date.now();
    
    // 解析 RSS
    const feed = await rssParser.parseURL(source.url);
    const items = feed.items.slice(0, config.rssFirstRunMaxArticles);  // 首次限制 50 篇

    // 标题去重
    const normalizedTitles = items.map(item => generateNormalizedTitle(item.title));
    const existing = await db.articles.findByNormalizedTitles(normalizedTitles);
    const newItems = items.filter(item => !existing.has(normalize(item.title)));

    // 批量插入
    await db.articles.batchInsert(newItems.map(item => ({
      rss_source_id: source.id,
      title: item.title,
      title_normalized: normalize(item.title),
      url: item.link,
      description: item.contentSnippet,
      content: item.content || item.contentSnippet,
      published_at: item.pubDate,
      source_origin: 'rss'
    })));

    // 更新 last_fetched_at
    await source.update({ last_fetched_at: new Date() });

    // 记录日志
    await db.rssFetchLogs.insert({
      rss_source_id: source.id,
      status: 'success',
      articles_count: items.length,
      new_articles_count: newItems.length,
      duration_ms: Date.now() - startTime
    });

    // 触发自动过滤 (异步，随机延迟)
    this.triggerAutoFilter(newItems.map(i => i.id));
  }

  private async triggerAutoFilter(articleIds: number[]) {
    const staggerMs = Math.random() * config.staggerDelayMaxMinutes * 60 * 1000;  // 0-30 分钟
    await sleep(staggerMs);
    
    for (const articleId of articleIds) {
      filterArticle(articleId, source.user_id).catch(err => {
        log.error({ articleId, error: err }, 'Auto-filter failed');
      });
    }
  }
}
```

### 2.8 用户认证与权限 (`api/auth.ts`)

**JWT 登录流程**：
```typescript
// POST /api/auth/login
app.post('/api/auth/login', async (req, res) => {
  const { username, password } = req.body;

  // 1. 查用户
  const user = await db.users.findUnique({ where: { username } });
  if (!user) return res.status(401).json({ error: 'Invalid credentials' });

  // 2. 验证密码 (bcryptjs.compare)
  const isValid = await bcrypt.compare(password, user.password_hash);
  if (!isValid) return res.status(401).json({ error: 'Invalid credentials' });

  // 3. 签发 JWT
  const token = jwt.sign(
    { userId: user.id, username: user.username, role: user.role },
    config.jwtSecret,
    { expiresIn: config.jwtExpiresIn }  // 默认 7 天
  );

  // 4. 返回
  res.json({ token, user: { id: user.id, username, role: user.role } });
});
```

**权限中间件**：
```typescript
// 必须登录
function requireAuth(req: Request, res: Response, next: NextFunction) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'No token' });

  try {
    const payload = jwt.verify(token, config.jwtSecret) as JwtPayload;
    req.user = { userId: payload.userId, role: payload.role };
    next();
  } catch {
    return res.status(401).json({ error: 'Invalid token' });
  }
}

// 只允许 admin
function requireAdmin(req: Request, res: Response, next: NextFunction) {
  if (req.user.role !== 'admin') {
    return res.status(403).json({ error: 'Admin only' });
  }
  next();
}
```

```typescript
// utils/crypto.ts
const algorithm = 'aes-256-cbc';

export function encrypt(text: string, keyHex: string): string {
  const key = Buffer.from(keyHex, 'hex');     // 32 字节
  const iv = randomBytes(16);
  const cipher = createCipheriv(algorithm, key, iv);
  const encrypted = Buffer.concat([cipher.update(text), cipher.final()]);
  return iv.toString('hex') + ':' + encrypted.toString('hex');  // 存储格式
}

export function decrypt(encryptedHex: string, keyHex: string): string {
  const [ivHex, dataHex] = encryptedHex.split(':');
  const key = Buffer.from(keyHex, 'hex');
  const iv = Buffer.from(ivHex, 'hex');
  const decipher = createDecipheriv(algorithm, key, iv);
  const decrypted = Buffer.concat([decipher.update(Buffer.from(dataHex)), decipher.final()]);
  return decrypted.toString();
}
```

### 5.2 语言检测与条件翻译

```typescript
// agent.ts (翻译模块)
function detectLanguage(text?: string): 'zh' | 'en' | 'unknown' {
  if (!text || text.trim().length === 0) return 'unknown';
  
  // 规则 1: 中文字符 Unicode 范围
  if (/[\u4e00-\u9fff]/.test(text)) return 'zh';
  
  // 规则 2: 英文字母占比 > 60% 且长度 ≥ 10
  const letters = (text.match(/[A-Za-z]/g) || []).length;
  const total = text.replace(/\s+/g, '').length;
  if (letters >= 10 && letters / Math.max(total, 1) > 0.6) return 'en';
  
  return 'unknown';
}

export async function translateArticleIfNeeded(
  title: string,
  content: string | undefined,
  userId: number
): Promise<{ titleZh?: string; summaryZh: string; sourceLang: string }> {
  const titleLang = detectLanguage(title);
  const contentLang = detectLanguage(content);
  
  // 仅当中文缺失且英文存在时翻译
  const shouldTranslateTitle = titleLang === 'en' && !/[\u4e00-\u9fff]/.test(title);
  const shouldTranslateContent = contentLang === 'en';
  
  if (!shouldTranslateTitle && !shouldTranslateContent) {
    return { summaryZh: '', sourceLang: contentLang };
  }
  
  // LLM 翻译调用
  const llm = await getUserLLMProvider(userId, 'translation');
  const prompt = `Translate the following ${contentLang === 'en' ? 'English' : '中文'} text to Chinese...`;
  const translated = await llm.chat([{ role: 'user', content: prompt }], {
    temperature: 0.3,
    label: 'translation'
  });
  
  return {
    titleZh: shouldTranslateTitle ? translated.title : null,
    summaryZh: translated.summary,
    sourceLang: contentLang
  };
}
```

### 5.3 相关文章增量刷新算法

**问题**：新文章处理完成后，如何让它出现在老文章的「相关文章」列表中？

**传统方案缺陷**：
- 每次新文章都触发所有老文章重新计算 → O(N²) 复杂度
- 对于 10,000 篇文章，每次刷新 10,000 次向量查询 → 不可行

**增量刷新方案**：
```typescript
// pipeline.ts (fire-and-forget)
processArticle(articleId, userId)
  .then(() => {
    // 新文章完成后，只刷新与新文章相似的老文章
    return incrementalRefreshRelated(articleId, userId, { topN: 10, minScore: 0.5 });
  });

// articles-refresh.ts
export async function incrementalRefreshRelated(
  newArticleId: number,
  userId: number,
  options: { topN: number; minScore: number }
): Promise<number[]> {
  // 1. 获取新文章的向量
  const newArticle = await db.articles.findUnique({ where: { id: newArticleId } });
  const newEmbedding = await getEmbedding(buildVectorText(newArticle), userId);
  
  // 2. 向量搜索，找出与此新文章相似的老文章（排除自身）
  // 注意：这里搜索的是其他文章向量，看是否匹配这篇新文章
  const hits = await queryVector(userId, newEmbedding, 100, {
    where: { user_id: userId, article_id: { $ne: newArticleId } }
  });
  
  const similarOldArticles = hits
    .filter(hit => hit.score >= options.minScore)  // 相似度阈值 0.5
    .slice(0, options.topN);                       // 最多 10 篇
  
  // 3. 对每个老文章，重新计算其相关文章列表
  // (这会包括这篇新文章，因为新文章与它是相关的)
  const refreshedIds: number[] = [];
  for (const old of similarOldArticles) {
    await refreshRelatedArticles(old.articleId, userId, 5);  // 重新计算
    refreshedIds.push(old.articleId);
  }
  
  return refreshedIds;
}
```

**效果**：新文章处理后，自动出现在与其语义相似的 10 篇老文章的「相关文章」列表中，而无需全量刷新。

### 5.4 混合搜索归一化策略

**问题**：向量相似度分数范围 [0,1]，关键词匹配分数 0-1，两者量纲不同，如何加权？

**解决方案**：
```typescript
// search-service.ts

// 策略 A: 归一化（搜索页面）

const maxSemScore = Math.max(...semanticResults.map(r => r.semanticScore), 0.01);
```

**实验结果**：

## 九、已知限制与改进方向

### 9.1 当前限制

| 限制 | 影响 | 临时方案 |
|------|------|----------|
| SQLite 并发写入瓶颈 | 多进程同时处理时 `SQLITE_BUSY` | WAL 模式 + systemd 单实例 |
| ChromaDB 数据未持久化 | 容器重启后向量丢失 | 挂载 volume 到 `data/vector/chroma` |
| LLM 翻译成本高 | 大量英文文献翻译费用累积 | 按需翻译（仅英文） + 缓存结果 |
| CNKI 反爬频繁 | 爬虫可能失效 | 定期更新 XPath / 使用 Playwright |
| 无全文 PDF 下载 | 仅能处理 RSS 摘要 | 集成 Paper PDF API（独立服务） |
| 单用户单租户 | 无多用户数据完全隔离（虽有 user_id） | 未来可加租户 ID 前缀 |

### 10.2 项目特色总结

1. **真正的 AI-Native 应用**：LLM 深度嵌入到数据处理的每个关键环节（过滤、翻译、洞察报告），而非简单的 API 封装。

2. **完整的数据生命周期管理**：从采集 → 过滤 → 翻译 → 向量化 → 搜索 → 通知 → 洞察，形成闭环。

3. **高度可配置**：几乎所有逻辑参数（过滤提示词、LLM 配置、主题词、调度时间）都可动态配置，无需改代码。

4. **生产级的错误处理**：重试机制、故障转移、降级策略、完整日志链路，适合 7×24 运行。

5. **清晰的代码架构**：Service 层、API 层、数据层分离，易于维护和扩展。

6. **多源数据融合**：RSS、期刊爬虫、关键词订阅三种来源统一处理，满足学术文献收集的多样化需求。
