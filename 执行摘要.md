## 一、项目一句话介绍

> 一个自动化学术文献追踪系统，使用 LLM 从 RSS、期刊官网、Google Scholar 等多源抓取文献，智能过滤无关内容，自动翻译并向量化，提供语义搜索和相关推荐。

## 三、技术架构亮点

### 3.1 核心架构原则

1. **AI-Native**：LLM 深度集成到数据流水线（过滤、翻译、洞察）
2. **渐进式处理**：流水线幂等性，失败只重试失败阶段
3. **故障转移**：LLM 多配置链式 fallback，搜索自动降级为关键词
4. **多租户隔离**：所有数据 `user_id` 隔离，ChromaDB Collection 分片
5. **配置驱动**：提示词、LLM 配置、调度时间均可 UI 动态配置

---

### 3.2 技术栈选择理由

| 技术 | 选型理由 |
|------|----------|
| **Node.js + TypeScript** | 生态成熟，类型安全，异步 IO 适合爬虫/调度 |
| **SQLite + Kysely** | 轻量部署，WAL 模式支持并发读取 |
| **ChromaDB** | 轻量向量数据库，支持持久化，Client 模式嵌入 |
| **OpenAI / Gemini** | 业界标准 LLM，成本可控（$0.001~0.01/篇） |
| **systemd** | Linux 原生进程管理，优雅关闭，自启动 |
| **Python venv** | 隔离爬虫依赖，避免 70+ AI 库污染主环境 |

## 四、关键设计决策

### 4.1 为什么选择 ChromaDB 而不是 Pinecone/Milvus？

**决策**：ChromaDB Client 模式（持久化到本地文件）

**理由**：
- ✅ 零运维：无需额外服务，直接嵌入应用
- ✅ 轻量级：适合 10 万篇以下文献规模
- ✅ 成本低：无需云服务费用
- ❌ 不足：单机部署，无分布式能力
- **适用场景**：个人/小团队使用 (< 10 万向量)

**扩展路径**：如果未来需要更大规模，可替换为 Qdrant（兼容 API）。

---

### 4.2 为什么使用 Python 子进程而不是 Node.js 直接爬取？

**决策**：Python 独立虚拟环境 + 子进程调用

**理由**：
- ✅ 爬虫库生态：Selenium、requests、BeautifulSoup、lxml 在 Python 更成熟
- ✅ 环境隔离：爬虫依赖（70+ 包）不影响主服务
- ✅ 维护简单：爬虫独立更新，无需重新编译 native 模块
- ✅ 错误隔离：Python 进程崩溃不影响 Node.js 主服务
- ❌ 不足：进程间通信（JSON 序列化）有开销

---

### 4.3 为什么使用 SQLite WAL 模式而不是 PostgreSQL？

**决策**：SQLite + WAL 模式（Write-Ahead Logging）

**理由**：
- ✅ 零配置：单文件，备份简单
- ✅ WAL 支持：读写不互斥，并发读取性能好
- ✅ better-sqlite3 原生模块：同步驱动，无 GIL 竞争
- ✅ 适合场景：个人/小团队（< 10 万文章，< 10 并发写入）
- ❌ 不足：写入并发有限（~5-10 写/s），不适合高并发写入场景

**扩容路径**：当文章数 > 50 万时，可迁移到 PostgreSQL + pgvector。

---

### 4.4 为什么流水线设计为 4 阶段而非单体函数？

**决策**：state machine + process_stages JSON 字段

**优势**：
1. **幂等性**：阶段状态记录，失败后重试只补缺失阶段
2. **可视性**：每阶段耗时、错误信息记录到article_process_logs
3. **灵活性**：新增阶段（如「全文 PDF 下载」）只需扩展 stages 字段
4. **调试友好**：`process_stages = {"markdown":"completed","translate":"failed",...}` 一目了然

---

### 4.5 为什么同时支持三种文章来源（RSS/期刊/关键词）？

**决策**：统一抽象为 Article 模型，来源字段 source_origin 区分

**好处**：
- ✅ 统一处理：过滤、翻译、向量化、搜索逻辑完全一致
- ✅ 可组合：用户可同时订阅 RSS + 期刊 + 关键词
- ✅ 可扩展：新增来源类型只需扩展 source_type 枚举，无需改核心逻辑

## 十、常见问题 FAQ

**Q1: 为什么过滤通过率这么低（10-30%）？**  
A: 这是正常现象。系统设计为**精确过滤**而非召回全部，只推送高度相关的文章。可通过调整 system_prompt 中的评分标准或降低 `minRelevanceScore`（默认 0.6）来提高通过率。

**Q2: 可以同时追踪多个领域吗？**  
A: 可以。在「设置 → 关注领域」中添加多个领域，每个领域配置不同的主题词和权重。LLM 会分别评估每个领域，只要有一个领域通过即视为通过。

**Q3: LLM 调用费用如何控制？**  
A: 主要通过以下方式：
- 使用廉价模型（gpt-4o-mini / gemini-1.5-flash）约 $0.001-0.003/篇
- 仅在英文文章时翻译（中文文章跳过翻译）
- 可配置 LLM 限流（60 请求/分钟）
- 可关闭翻译功能（仅过滤）

**Q4: 数据备份和迁移是否方便？**  
A: 非常方便。SQLite 数据库是单文件，直接复制 `data/rss-tracker.db` 即可。向量数据在 `data/vector/chroma` 目录，一并复制即可完整迁移。

**Q5: 能否部署在 Docker 里？**  
A: 可以，但项目原生设计为 bare-metal systemd 部署。如需 Docker，建议：
- 主应用：`node:24-alpine` 镜像
- ChromaDB：`chromadb/chroma` 官方镜像
- 数据卷挂载：`-v /opt/lis-rss-daily/data:/app/data`
- 使用 docker-compose 编排

**Q6: 支持多用户吗？数据如何隔离？**  
A: 支持多用户（SQLite `users` 表），但**非 SaaS 多租户**。多用户共享同一数据库，通过 `user_id` 字段隔离数据（所有表都有 user_id）。适合个人或小团队，不适用于公开服务。

**Q7: ChromaDB 数据持久化吗？重启后向量会丢失吗？**  
A: ChromaDB 默认持久化到 `data/vector/chroma` 目录。确保配置了 `--path` 参数（部署脚本已配置），重启服务后向量数据保留。

**Q8: 如何升级依赖版本？**  
A:
```bash
cd /opt/lis-rss-daily
pnpm update            # 更新 Node.js 依赖
source lis-rss/bin/activate
pip install --upgrade chromadb   # 更新 Python 依赖
```
